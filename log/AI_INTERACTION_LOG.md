### Phase 4: Metrics and Data Cleaning

**User**: Що саме аналізуємо? Просто кількість тривог?

**Assistant**: Кількість тривог - не найкраща головна метрика. Одна тривога може тривати 10 хвилин, а інша 6 годин. Тому головна метрика має бути `total_duration_min` за день.

Додатково:

- `alert_count` - кількість окремих тривог.
- `avg_duration_min` - середня тривалість.
- `max_duration_min` - найдовша тривога за день.
- `night_share` - частка тривог у нічний час.
- `time_bucket` - night/morning/afternoon/evening.

**User**: А ніч як рахувати?

**Assistant**: Пропоную ніч як 23:00-06:00 за Europe/Kyiv. Це зрозуміло і релевантно для повітряних тривог, бо нічні атаки мають окремий соціальний і психологічний ефект.

У коді це буде так:

```python
if 23 <= hour or hour < 6:
    return "night"
elif 6 <= hour < 12:
    return "morning"
elif 12 <= hour < 18:
    return "afternoon"
else:
    return "evening"
```

**User**: А як з аномаліями?

**Assistant**: Треба прибрати технічні аномалії, але не викидати реальні великі тривоги просто тому, що вони великі. Я зроблю два фільтри:

1. Базовий фільтр: тривалість має бути більше 0 і менше 24 годин.
2. Аномалії: top 1% дуже довгих записів плюс IQR-фільтр окремо по містах.

Це дає баланс: ми прибираємо очевидно підозрілі записи, але не чистимо дані настільки агресивно, щоб зникла сама воєнна реальність.

**User**: Це треба якось пояснити в README?

**Assistant**: Так, коротко. У README можна написати, що raw event data було агреговано до daily time series, а екстремальні технічні записи відфільтровано через duration constraints, percentile threshold та city-level IQR. Це звучить професійно і не перевантажує.

Можна згадати, що `(1,1,1)` використано як conservative baseline. Головна цінність не в тому, що ARIMA ідеально прогнозує, а в тому, що ми порівнюємо кілька класів моделей і чесно показуємо їхні обмеження.
